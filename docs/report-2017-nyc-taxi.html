Ibis Benchmark
  • 2017 Yellow Taxi Trip Dataset

2017 Yellow Taxi Trip Dataset

Objective

This benchmark aims to show the performance in time for some expressions using Ibis with OmniSciDB and Pandas

Terminology
  • IPC: Inter Process Communication. OmniSciDB uses Apache Thrift for IPC.
  • Cursor: Cursors are used by database programmers to process individual rows returned by database system queries.
  • CPU: Central Processing Unit.
  • GPU: Graphics Processing Unit.
Dataset

This benchmarking uses the 2017 Yellow Taxi Trip Dataset

The dataset has:

  • Rows: 113M
  • Columns: 17

Methodology

This benchmark measures the execution time (in seconds) consumed by expressions using Ibis OmniSciDB backend and pure Pandas. Ibis OmniSciDB backend can be used using three differents execution types: IPC CPU, IPC GPU and Cursor. The expressions used by this benchmark are:

  • t.trip_distance.max()
  • t.trip_distance.min()
  • t.trip_distance.mean()
  • t.trip_distance.std()
  • t.trip_distance.sum()
  • t[t.fare_amount < 100].fare_amount.mean()
  • t[t.fare_amount < 100].fare_amount.std()
  • t[t.fare_amount < 100].fare_amount.sum()
Where t is the 2017 Yellow Taxi Trip Dataset. The time reported here doesn't considerer the time for data loading, eg. Pandas read_csv or OmniSciDB COPY FROM

For more information about the benchmark, check it out its source code here.

Results
Summary

Some considerations about this benchmark:

  • Ibis OmniSciDB backend, using cursor execution type, presents a better performance than Pandas and other Ibis OmniSciDB execution types for all expressions.
  • Ibis OmniSciDB backend using IPC GPU presents a better performance than using IPC CPU for all expressions.
  • Pandas presents a better performance than Ibis OmniSciDB backend using IPC execution types for simple expressions.
